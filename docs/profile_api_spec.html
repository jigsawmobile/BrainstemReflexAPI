Overview

This document describes the backend API usage inferred from ProfileController.swift. It focuses on the controller’s server-facing data-sharing features rather than local UI, export, or HealthKit-only operations.

The client implements these authenticated operations:

  • registering a device to the authenticated user
  • creating a device-specific data share
  • stopping data sharing
  • deleting uploaded user records
  • sharing device data to a backend wallet address
All protected endpoints rely on an access token obtained by WalletSettingsTableController. When the backend returns { "msg": "Token has expired" }, the client refreshes the token and retries by using follow-up actions.

Base URL

{HOST_URL}

Authentication Dependency

ProfileController does not implement its own login flow. It depends on the wallet controller for:

  • current access_token
  • token refresh
  • wallet address validation / restoration

Protected requests therefore use:

Authorization: Bearer <access_token>

Endpoints

1. Register Device

POST/api/v1/devices/register

Purpose

Registers a submitted device identifier for the authenticated user.

Headers

Authorization: Bearer <access_token>
Content-Type: application/json

Request

{
  "device_id": "submitted-device-id"
}

Known reply mapping inferred from client

StatusCodeMeaning
success1Device registered successfully; client proceeds to create data share.
success2Device already exists and is already registered to this user.
failed3Device does not exist.
failed4Device exists but registered to another user.

Example success-style reply

{
  "status": "success",
  "code": 1,
  "messages": "Device registered successfully"
}

Token expiration

{
  "msg": "Token has expired"
}

2. Create Data Share

POST/api/v1/metadata/create-data-share

Purpose

Creates the server-side data-sharing entity for the current device/user pair.

Headers

Authorization: Bearer <access_token>

Request body

No request body is sent by the client.

HTTP 200 success response

{
  "result": {
    "data_nft_address": "0x...",
    "data_share_id": "share-id",
    "data_token_address": "0x..."
  }
}

HTTP 400 failure response

{
  "messages": ["..."]
}

HTTP 401 token expiration

{
  "msg": "Token has expired"
}
On success, the client stores data_nft_address, data_share_id, and data_token_address, then marks token status as READY.

3. Stop Data Sharing

POST/api/v1/metadata/stop-data-share

Headers

Authorization: Bearer <access_token>

Request body

No request body is sent by the client.

Success response

{
  "result": "success"
}

Validation / business error

{
  "messages": ["..."]
}

Token expiration

{
  "msg": "Token has expired"
}

4. Delete User Records

POST/api/v1/delete-records

Headers

Authorization: Bearer <access_token>

Request body

No request body is sent by the client.

Success response

{
  "message": "Records deleted"
}

Token expiration

{
  "msg": "Token has expired"
}
The client only treats the literal message "Records deleted" as confirmed success.

5. Share Data to Backend Wallet

POST/api/v1/metadata/share-data?to_address=<wallet>&data_share_id=<share-id>

Purpose

Shares an existing device data share to a specific wallet address. The client uses this fixed backend wallet address:

0xAC27FC669b9fEB045bDdFc6ec4EdC84608CF9666

Headers

Authorization: Bearer <access_token>

Query parameters

ParameterDescription
to_addressTarget wallet address to receive the data share.
data_share_idIdentifier previously returned by create-data-share.

HTTP 200 success response

{
  "result": {
    "data_share_id": "share-id",
    "mint_tx": "transaction-id-or-hash"
  }
}

HTTP 400 failure response

{
  "messages": ["..."]
}

HTTP 401 token expiration

{
  "msg": "Token has expired"
}
The client refuses to call this endpoint until it already has a local data_share_id. If none exists, it treats the device as unregistered.

Token / Device Status

StatusMeaning in client
NO DEVICENo registered/usable device state is available.
PROCESSINGDevice/data-share creation request is in progress.
READYDevice and data-share flow completed successfully.

Error Handling

Standard token expiration contract

{
  "msg": "Token has expired"
}

Common message-array failure shape

{
  "messages": ["..."]
}

Mixed response conventions

The backend responses observed by this client are not fully uniform. Across endpoints, the client expects a mix of status + code + messages, result as string or object, message as string, and msg for token-expiration signaling.

Implementation Notes

  • Authentication is delegated to the wallet controller.
  • The class also supports local export of TXT, JSON, and HealthKit archives, but those are not backend API calls.
  • After successful device registration or data-share creation, the client may trigger additional peaq-related linking outside this controller.
  • For “share to wallet”, the target address is hard-coded in the client as a backend wallet address.
  • A normalized backend response schema would make this API easier for external integrators.
